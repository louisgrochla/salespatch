# Visual-QA audit + improvement proposals

Audit run 2026-05-16 against the 14 existing demos under `~/Desktop/salespatch-demos/`.

## TL;DR

The shipped visual-QA is sound architecturally (render → 3-layer vision → canonical JSON, dual implementation, parity contract) and demonstrably catches the headline bug class it was built for. But running it across the cohort exposed a more interesting picture: the *Bouquet Bar* readability failure is an outlier — most demos use text-on-solid-background heroes and avoid that class entirely. The bugs they DO have (hardcoded "live" features, CTA hierarchy collapse, missing hero CTAs) are caught poorly or not at all by Layer 1 as written. Bigger leverage now is in **expanding what visual QA looks for and using its output to make `/build-demo` self-correct**, not in adding more sophistication to the existing layers.

## Section 1 — Audit findings

### What works

1. **Renderer is reliable**: All 14 demos rendered cleanly (3-8s each, no flakes). Hero PNGs 99-794KB, full-page 1.2-7.9MB. Headless Chromium + the 1.2s Google-Fonts wait + 2× device-scale produces high-fidelity screenshots that vision can read precisely.

2. **Layer 1 catches its target bug class**: text-over-image readability on the Bouquet Bar hero is flagged critical, with location + cause + WCAG-equivalent contrast estimate. Static QA scored 100/100 on the same demo — the layer is doing the job static can't.

3. **Layer 2 grading produces useful drift signal**: 5-dimension brand-fidelity scoring (palette/typography/logo/positioning/signature) returns coherent per-dimension drift notes that map directly to fixable issues. The Bouquet Bar grade (4.6/5) felt honest — palette 4 because the hero is photo-dominated rather than the brief's 60% dove grey, every other dimension 4 or 5.

4. **Layer 3 owner-reaction is the right kind of qualitative signal**: returns recognition + first reaction + pushbacks + would_buy + test_of_success pass/fail. The Bouquet Bar reaction surfaced three concrete pushbacks ("where does the button go", "what about Google", "grey is cold") that the build process can't surface from CSS alone.

5. **Dual implementation contract holds**: identical `VisualQaResult` schema across manual + SDK paths. NERVE-side ingest doesn't care which producer ran. Drift is impossible by construction because both paths import the same prompt module.

### What's broken or under-built

#### A1. The renderer-correctness pass surfaced a real workflow bug

When the audit started, all 13 non-Bouquet-Bar demos showed 0KB PNGs. Root cause: I'd switched back to `main` after opening the PR. `apps/nerve/scripts/qa-visual-render.ts` doesn't exist on `main`. The `npx tsx` call silently failed because my error-handling stripped the "qa-visual-render" line from stderr.

Two implications:
- The `/build-demo` skill text references the script by absolute path, which means it'll silently fail in any session run from `main` until PR #93 merges. Skill text should add a `[ -f "$SCRIPT" ] || { echo "skipped (script missing)"; exit 0; }` guard around the render step, mirroring the existing static-QA guard.
- My loop's `2>&1 | tail -1` pattern hid the error. The `qa-visual-render.ts` script should write a `.qa-visual/render-result.json` on success so a caller can probe outcome without parsing stderr.

#### A2. Hardcoded "live" features hide credibility bombs

Audit hypothesis: noose-and-needle shows "Today · Wed 7 May" and an artist availability list ("Letitia walk-ins from 12", etc.) in the hero. Looks impressive in the screenshot. Visual QA, looking at the rendered pixels only, has no way to tell this is **hardcoded into static HTML**, not generated by `new Date()`. Grep of the rendered demo confirms: zero dynamic-date logic, the strings are baked.

Cohort check:
| Demo | "Live" claim in hero | Actually dynamic? |
|---|---|---|
| noose-and-needle | "Today · Wed 7 May" + 6-artist availability list | NO — hardcoded |
| fable | "OPEN TODAY · UNTIL 5PM" green-dot pill | NO — hardcoded |
| cafe-100 | "CLOSED · BACK AT 8:30" pill | YES — uses `getDay()` + `getHours()` |
| bouquet-bar | none claimed | N/A |
| blackbird | none claimed | N/A |

**Two-thirds of the "live status" features in this cohort are theatre.** The rep opens the demo on Friday, the owner sees "Today · Wed 7 May", the rep loses the room. Visual QA in its current form is blind to this. The right detector is a **hybrid pass**: vision identifies what *looks* live (date pills, "today" labels, queue counters, items-remaining ribbons), then static AST/JS-scan checks whether the corresponding text is produced by a date/time call. Mismatch = critical credibility bug.

#### A3. Missing-hero-CTA pattern slips through silently

The Cult of Coffee hero has **no primary CTA at all** — the headline + a "CLOSED · BACK TOMORROW 8:30" status badge, but nothing to tap. The reader cannot do anything from above-the-fold. The Layer 1 prompt currently asks for "above-the-fold CTA presence" as one of the 6 bug categories. Whether it would flag this would depend on the vision model interpreting "BACK TOMORROW 8:30" as a status (correctly) rather than a CTA. The prompt should be more explicit: "an above-the-fold primary CTA must be a tappable verb-led action, not a status indicator".

#### A4. CTA hierarchy collapse not graded

- noose-and-needle has the same CTA twice (nav + hero), both labelled "FIND YOUR ARTIST →" with the same gold styling. That's redundancy, not hierarchy.
- fable has three primary-weight CTAs ("RESERVE A SEAT" / "SEE WHAT'S ON" / "FIND THE DOOR") competing in the same viewport.
- cafe-100 has a balanced two-CTA hierarchy (red filled "See the wraps →" + outlined "100 Holburn Street").
- Bouquet-bar has a balanced two-CTA hierarchy.
- The "what makes a good CTA hierarchy" judgement currently lives nowhere in the prompts. It's a meaningful conversion-craft signal Layer 1 should grade.

#### A5. Below-the-fold quality is under-graded

The render captures `full.png` (fullPage scroll) AND `hero.png`. Both are sent in every layer's vision call. But the prompts are hero-biased — they explicitly direct attention to "above-the-fold" and treat the full-page screenshot as a sanity check rather than a structured pass. Sections 2 through N of every demo get hand-waved.

Observable below-fold weaknesses in the cohort that the current prompts likely miss:
- Lookbook grids with uneven tile sizes / awkward whitespace
- Footer slammed against the previous section with no breathing room
- About sections with placeholder copy that doesn't match the brand voice
- Map iframes filtered grey but with default Mapnik styling fighting the page palette

A dedicated "scroll-through narrative" mini-layer would help: 4-5 sentences summarising whether the page tells a coherent story top to bottom, and grading section-to-section consistency.

#### A6. No photo-quality grading

The build picks photos by role (logo/storefront/product_close/etc.) but never grades whether the chosen photo is *technically good* (in focus, well-composed, well-lit, suitable for hero size). A blurry phone snap of a wedding bouquet can pass role-matching and end up as the hero. Vision is well-suited to grade this; current prompts don't.

#### A7. Voice consistency between rendered copy and brief.voice_quotes isn't checked

Layer 2 grades typography (do the fonts match the brief) but not whether the actual *words* on the page match the brief's `voice_quotes[]`. If the brief committed to "Beautiful flowers for all occasions" + "DM to contact or enquire 🩷" + "Pastels with a pop" but the rendered hero says "Welcome to The Bouquet Bar, your premier florist" — vision would grade typography 5 (fonts are right) and miss the voice drift entirely.

#### A8. Mobile-only viewport

The rendering is 375×812 (iPhone 13 mini). Reps pitch on iPhones — correct. But many owners ask "can I see it on my computer?" after the pitch lands. Demos that work at 375 may break at 1280. No safety net.

### Code-level findings

Reading the shipped scripts with critical eyes:

1. **`qa-visual.ts`** runs three separate SDK calls. Better for focus, worse for cost (3× per demo). Defensible at ~£0.005/demo but the cost-conscious version should expose a `--combined` flag that does one call with structured output covering all three layers.

2. **`qa-visual.ts` error handling exits non-zero on any layer failure**, losing whatever partial result was already computed. Better: write the partial result file with the failed layer set to `null`, surface the failure, exit 0 so the calling skill can still continue with what's known.

3. **`qa-visual-render.ts` waits a fixed 1200ms for fonts**. Network-flaky. Should use `page.waitForLoadState('networkidle')` plus a `page.evaluate(() => document.fonts.ready)` promise. Deterministic vs the current wall-clock guess.

4. **`max_tokens: 2000` hardcoded** in the SDK runner. Sufficient for Layer 1 and 2 but Layer 3's owner reaction can run over when the model gives a rich response. Should be 3500 minimum.

5. **`buildOwnerReactionUserMessage` falls back to "you are the owner of X"** when `ownerName` is null. Stronger: "you are the owner of X (a sole trader running this from a home studio)" — pull additional persona context from the brief's `business_type` and `enrichment.companies_house` data. The richer the persona, the less generic the reaction.

6. **No retries on transient API failures**. A 503 from Anthropic shouldn't kill the entire pass. One retry with backoff is cheap insurance.

7. **The `.md` spec says "drift between .ts and .md is forbidden"** but there's no test enforcing it. A trivial CI check (or a `npx tsx --check` step) that grep-counts the prompt name occurrences in both files would catch silent drift.

8. **`producer` and `model` fields are the only difference between manual and SDK output** — but the manual-flow skill text doesn't enforce that the in-session producer string literally equals `"manual_skill"`. A schema-validation step at write time (Zod, ajv, or a hand-rolled check) would prevent shape drift.

## Section 2 — Improvement proposals, ranked

Scored on `(impact × cohort coverage) / effort`. Impact is how much a fix moves close rate; cohort coverage is how many demos benefit; effort is engineering days. Anything HIGH ships in the next 1-2 PRs.

### HIGH-impact, LOW-effort

**1. Hybrid dynamic-content honesty check (Layer 1 extension + static JS scan).** Catches the noose-and-needle / fable hardcoded-date class of bug. Vision identifies what *looks* live in the hero (date pills, today labels, queue counters, items-remaining ribbons); a small new `qa-visual-dynamic.ts` script greps the JS for `getDay`/`getDate`/`getHours`/`toLocaleDateString`/`new Date(` and confirms each candidate is wired to one of them. Mismatch = critical. **Effort: 4 hours. Impact: prevents the most credibility-damaging demo failure I found.**

**2. Extend Layer 1 prompt with CTA-hierarchy + missing-CTA-status-confusion rules.** Two new bug categories: "redundant CTA pairs in the hero" and "the above-the-fold CTA is a status indicator not a verb-led action". Catches the noose-and-needle redundancy + Cult of Coffee blind spot. **Effort: 1 hour (prompt edit). Impact: cohort-wide CTA discipline.**

**3. Add Layer 4: voice-quote consistency.** Vision reads the rendered headlines + body copy; spec passes `brief.voice_quotes[]` as expected phrases; vision grades whether the rendered copy preserves the brief's voice or has drifted to generic. **Effort: 2 hours (new prompt + interface field). Impact: catches the build dropping the brief's verbatim language, which kills the "wait, that's literally my Facebook bio" reaction.**

**4. Make the render write `.qa-visual/render-result.json`.** Callers (skill text, future automation) can probe success without parsing stderr. Also include heroPath/fullPath/heroPx/fullPx/durationMs for downstream queries. **Effort: 30 min. Impact: prevents the silent-failure pattern I hit in the audit.**

**5. Render at desktop viewport too (1280×800).** Capture `hero-desktop.png` + `full-desktop.png`. Send both pairs to layers when present. Catches the responsive break I can't currently see. **Effort: 30 min in renderer + prompt note. Impact: closes the laptop-pivot blind spot.**

**6. Skill-text guard around the render step.** `[ -f "$SCRIPT" ] || { echo "Visual-QA: skipped (script missing)"; exit 0; }` mirror of the existing static-QA pattern. **Effort: 5 min. Impact: graceful degradation when running from a branch missing the scripts.**

### MEDIUM-impact, LOW-effort

**7. Photo-quality grading (new mini-layer or extend Layer 2).** Vision rates each photo embedded in the demo on a 1-5 scale for focus / lighting / composition / suitability-for-role. Surfaces the "blurry phone snap as hero" failure. Already partially served by Layer 2 brand-fidelity grading but currently roles-up; should be per-photo. **Effort: 2 hours. Impact: prevents the easy-to-make build mistake of picking the first photo for the role rather than the best.**

**8. Customer-reaction simulation (Layer 5).** Mirror of Layer 3 but role-playing the *customer* who landed from a Google search. "You searched 'Aberdeen florist' and clicked this. Within 5 seconds, would you book?". Different signal from owner-reaction. The owner reacts to "is this me?"; the customer reacts to "should I trust this?". **Effort: 2 hours. Impact: catches demos that satisfy the owner but won't convert traffic.**

**9. Section-by-section grading on the full-page screenshot.** Slice the fullPage PNG into N section-height chunks; have vision grade each section's design rhythm, whitespace, copy density, brand consistency. Returns an array. **Effort: 3 hours (Playwright section-bounding + prompt + parsing). Impact: surfaces below-the-fold quality I'm currently hand-waving.**

**10. Auto-fix loop in `/build-demo`.** When Layer 1 flags `severity=critical`, the skill auto-attempts a known-good remedy (e.g. for "text-over-image low contrast" → bump gradient overlay opacity to 0.6 at top stop), re-renders, re-runs visual QA. Iterates 2-3 times before giving up. Closed-loop quality. **Effort: 4-6 hours. Impact: turns visual QA from "report card" into "guardrail".**

### MEDIUM-impact, MEDIUM-effort

**11. NERVE `/api/ingest/qa-visual-result` route + Prisma model.** Currently posts 404. Without it the warehouse can't answer "do high-visual-QA demos close better?". **Effort: 3 hours. Impact: enables every downstream cohort-comparison query.**

**12. Cohort-aware baselines.** Once N ≥ 10 closed demos exist per vertical, NERVE returns "for vertical=barber, median Layer-2 brand-fidelity grade was 4.3 on closed demos, 3.8 on rejected". Each new demo's visual QA includes a "below cohort baseline" flag per dimension. **Effort: 4 hours (after NERVE route ships). Impact: turns QA from absolute to relative grading.**

**13. A/B variant scoring.** `/build-demo` optionally produces 2-3 hero variants. Visual QA scores all three. Build ships the winner. **Effort: 6-8 hours (variant generation + multi-render + selection). Impact: structurally lifts the floor on every demo.**

**14. Pre-launch checklist as structured output.** Extract every "TODO before live" from the demo + brief (real reviews to lift, real phone to wire, form backend, etc.) into a JSON checklist. The rep shows the owner during the pitch so the £350 isn't seen as "the whole site shipped now". **Effort: 2 hours. Impact: pitch-clarity tool, soft credibility lift.**

**15. Comparison render vs the competitor sites named in the brief.** Render the actual sites of Anastasia / Bauers / Flower Vogue at 375×812. Vision pass: "which of these four would a customer trust?". **Effort: 4 hours. Impact: turns the visual QA into a competitive proof point for the rep ("yours scored higher than Anastasia").**

### LOWER-priority (worth tracking, not next)

**16. Performance metrics from Playwright** (LCP/CLS/TTI). Useful long-term but doesn't move the first-impression close-rate dial.

**17. Form-submission simulation via Playwright.** Fills the enquire form and verifies the mailto fires. Catches broken form wiring. Mostly low value once the manual-test step exists.

**18. Cross-browser render** (Safari / Firefox). Worth checking but Chromium covers the vast majority of pitch-time devices.

**19. Real WCAG luminance math via axe-core.** Replaces vision-estimated contrast with deterministic numbers. Useful only on borderline cases; vision is currently good enough at "obviously failing" and "obviously fine".

**20. Demo-to-final drift tracker.** When a sold demo gets handed to fulfilment, capture the final shipping page and visual-QA against the original. Catches "the rep showed it, the dev rebuilt it, the owner got something different". Real once fulfilment volume picks up.

**PR-A — `feat(qa-visual): hybrid dynamic-content honesty + CTA hierarchy + voice-quote layer`**

- Layer 1 prompt extension for CTA hierarchy + status-as-CTA (proposal 2)
- New Layer 4 voice-quote consistency (proposal 3)
- Renderer writes `render-result.json` (proposal 4)
- Skill-text script-exists guard (proposal 6)
- Total: ~8 hours engineering. Catches the most credibility-damaging cohort failures.

**PR-B — `feat(qa-visual): desktop viewport + photo-quality grading + auto-fix loop`**
- Desktop viewport render (proposal 5)
- Per-photo quality grade (proposal 7)
- Closed-loop auto-fix on critical bugs (proposal 10)
- Total: ~6-8 hours. Turns visual QA from report to guardrail.

NERVE ingest route (proposal 11) is the long-lead unblocker for the cohort-baseline work and is the natural follow-up to PR-A or PR-B.
