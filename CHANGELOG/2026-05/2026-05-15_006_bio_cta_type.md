# lead_profiles — bio_cta_type structured field

## What changed

- `apps/nerve/prisma/schema.prisma` — `LeadProfile.bioCtaType` (`String?`,
  `bio_cta_type`). Composite index on `(vertical, bio_cta_type)` for
  per-vertical CTA-type distribution queries.
- `apps/nerve/prisma/migrations/21_bio_cta_type/migration.sql` —
  `ALTER TABLE lead_profiles ADD COLUMN bio_cta_type TEXT` + the index.
- `apps/nerve/src/lib/sl-mas/leadProfileStore.ts` — field added to
  `LeadProfileInput`, `LeadProfileRow`, and both mappers.
- `apps/nerve/src/app/api/ingest/lead-profile/route.ts` — hard
  validation: `bio_cta_type` must be one of `call | dm | link_in_bio |
  fresha | booksy | treatwell | website | none`. Anything else returns
  400 with the allowed list spelled out in the error.
- `~/.claude/skills/spec-site-brief/SKILL.md` (user-level) — Phase 1
  Verify gains a "Bio CTA type" bullet listing all eight values with
  discriminating descriptions and the tie-break heuristic ("when in doubt,
  prefer the higher-friction option because that's where the booking
  actually happens"). `lead-profile.json` schema example carries the
  field at top level.

## Why

The Instagram bio's call-to-action is the single most predictive signal
for which diagnosis applies. Blackbird Bakery's bio reads "No DMs (Call
us/ pop into the Bakery)" — that ONE phrase tells the qualifier:

1. Phone-led service model, not DM-led
2. Any pitch that proposes replacing the phone will fail (owner has
   explicitly opted out of that workflow)
3. Diagnosis leans "discovery failure" or "trust gap", not "booking
   friction"

We currently extract this in prose during Phase 1 verification. Lifting
it to a structured `lead_profiles` column lets:

1. The AI layer learn "for vertical=bakery, leads with `bio_cta=call`
   close at X%; leads with `bio_cta=dm` close at Y%" without
   re-judging every brief.
2. The qualifier auto-set `avoid_topics` based on the CTA type
   (`call` → "don't pitch replacing the phone"; `dm` → "don't pitch
   eliminating the inbox").
3. Future agents short-circuit diagnosis where the CTA gives it away.

## Stack
Next.js 14 + Prisma 5 + PostgreSQL. Producer-side judgement call inside
`/spec-site-brief` Phase 1 — no new actor, just structured extraction
from research the brief writer already does.

## Integrations
- NERVE `/api/ingest/lead-profile` extended (HMAC unchanged).
- No new external dependencies. Zero additional Apify cost.

## How to verify

1. `cd apps/nerve && npx prisma generate && npx tsc --noEmit` — clean
   (verified locally).
2. `cd apps/nerve && npx prisma migrate deploy` — applies migration
   `21_bio_cta_type`.
3. Run `/spec-site-brief` on a fresh lead. Verify:
   - `outputs/lead-profile.json` carries `bio_cta_type` at top level
     with one of the eight values
   - NERVE returns HTTP 200, field stored
4. POST a payload with `bio_cta_type: "phone"` (typo) — confirm 400 with
   `"bio_cta_type must be one of: call|dm|link_in_bio|fresha|booksy|treatwell|website|none"`.
5. Query the warehouse for per-vertical distribution:
   ```sql
   SELECT vertical, bio_cta_type, COUNT(*) AS n
   FROM lead_profiles
   WHERE bio_cta_type IS NOT NULL
   GROUP BY vertical, bio_cta_type
   ORDER BY vertical, n DESC;
   ```

## Known issues

- Existing rows (Blackbird, Nevermind, The Cult of Coffee, etc.) get
  NULL on apply. Forward-only — producer fills on next
  `/spec-site-brief` run for each lead. The SL-MAS replay-to-refresh
  pattern.
- "Adding a new CTA value later" requires a one-line route-validator
  change (extend the `known` Set). No migration needed, but the value
  must be added in three places to stay in sync: route validator,
  migration 21's known-values comment, and SKILL.md Phase 1 bullet.
  The route validator IS the source of truth; the others are docs.
- The "primary CTA when two are present" tie-break is judgement. The
  SKILL.md heuristic ("prefer higher-friction") is a sensible default
  but the brief writer can override based on diagnosis intent. Over
  time we can compare close rates of `call` vs `dm` vs `link_in_bio`
  leads and revisit the heuristic empirically.
