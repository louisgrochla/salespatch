# F2 fix — NERVE public demo URL wins over pitch_brief value on import

## What changed
- `apps/sales-dashboard/src/app/api/admin/import-from-nerve/route.ts` —
  flipped the `demo_site_domain` precedence in `buildNotes()`. Was:
  ```
  pitch?.demo_site_domain ?? buildDemoUrl(b.slug, !!demo)
  ```
  Now:
  ```
  buildDemoUrl(b.slug, !!demo) ?? pitch?.demo_site_domain ?? null
  ```
  The NERVE public route is always preferred when a demo_artefact exists
  in NERVE — that's where the actual HTML lives. The pitch_brief field
  is only used as a fallback for the legacy manual-upload flow.

## Why
The Cult of Coffee was the second lead pushed through the F2 pipeline.
Its `pitch_brief.demo_site_domain` was set to a placeholder subdomain
(`the-cult-of-coffee.salespatch.co.uk`) by the `/lead-json` skill — a
real URL would have required the skill knowing about NERVE's public
demo route, which it didn't.

The F2 import handler then preferred the (broken) pitch_brief value
over the (working) NERVE URL. When the salesperson opened the lead in
the iOS app, `ClientPresentationView.swift:432-441` prepended `https://`
to the bare domain and the WebView loaded
`https://the-cult-of-coffee.salespatch.co.uk` — a subdomain Vercel
catches with its password-protection screen. SP saw a black screen
asking for a code.

The fix flips the precedence so NERVE's public route always wins when
a demo exists. The placeholder subdomain becomes harmless dead data in
the pitch_brief — it's never read for the URL again. The `/lead-json`
skill markdown has been updated separately (user-level) to default to
the full NERVE URL going forward.

## Stack
- One conditional flip in TypeScript. No new dependencies. No schema
  change. Pure data-flow correction.

## Integrations
- iOS WebView (`ClientPresentationView.swift`) — receives the corrected
  URL via `notes.demo_site_domain`. No change needed on the iOS side;
  it was always doing the right thing with whatever URL the data
  contained.
- F2 queue → import → assign flow is the only path this affects. The
  legacy manual `/api/admin/leads` POST is unaffected — it never had a
  pitch_brief in scope.

## How to verify
1. `cd apps/sales-dashboard && npx tsc --noEmit` — clean (one
   pre-existing `resend` not-found in `src/lib/email.ts`, unrelated).
2. After Vercel deploy:
   - Find a new fresh lead (or reject + re-import an existing one).
   - Confirm `lead_assignments.notes.demo_site_domain` in Supabase ==
     `https://nerve.salespatch.co.uk/api/public/demo/<slug>` regardless
     of what the pitch_brief's value was.
   - SP opens the lead on the phone → demo renders.

## Known issues
- The Cult of Coffee assignment in Supabase still has the bad
  `the-cult-of-coffee.salespatch.co.uk` value because that row was
  inserted BEFORE this fix shipped. A one-off Supabase SQL update fixes
  it retroactively:
  ```sql
  UPDATE lead_assignments
  SET notes = jsonb_set(notes::jsonb, '{demo_site_domain}',
    '"https://nerve.salespatch.co.uk/api/public/demo/the-cult-of-coffee"')::text
  WHERE lead_id = 'the-cult-of-coffee';
  ```
- Future spec-site demos generated by the updated `/lead-json` skill
  will write the NERVE URL directly into `pitch_brief.demo_site_domain`,
  at which point the precedence flip becomes belt-and-braces. Belt is
  fine, belt with braces is better.
