# spec-site-brief — Google Maps photos in Phase 1.5

## What changed

- `~/.claude/scripts/apify/google-maps.sh` (user-level, not in repo) —
  replaced the dead HTTP-API stub with an MCP-era downloader. New signature
  `google-maps.sh <slug> <dest_dir> [items.json]`. Accepts both the place-
  wrapper shape (`[{placeName, photos[]}]`) and a flat photos array. Dedupes
  by `photoId` across categories. Filenames: `gmaps_<slug>_<photoIdShort>.jpg`.
- `~/.claude/skills/spec-site-brief/SKILL.md` (user-level, not in repo) —
  - Phase 1 verify: new bullet asking the brief writer to capture the
    canonical Google Maps URL (with CID) alongside the Google rating
    search.
  - Phase 1.5 photos: new "Google Maps photos (via Apify MCP)" sub-section,
    inserted BEFORE the existing Instagram block. Documents the
    three-step pattern (call `solidcode/google-maps-photos-scraper` →
    save items to `.cache/gmaps-items.json` → run `google-maps.sh`).
    Covers cost (£0.005/lead), category meanings (`by_owner` / `vibe` /
    `menu` / etc.), and how to use the category in Phase 2's photo role
    mapping.

## Why

The end-to-end chain test on Blackbird Bakery (2026-05-15) surfaced the
gap: Apify IG + Fresha + Mapillary cover the bakery's cake photography but
miss `storefront` and `interior` because Blackbird doesn't post those on IG.
Smoke-testing `solidcode/google-maps-photos-scraper` on Blackbird returned
3 owner-uploaded photos in under a second from a listing with **4,458 total
photos available** — none of which would have been reachable through the
existing photo sources.

Google Business Profile photos also surface the `uploadedByOwner` boolean
and Google's own category labels (Menu / Food & drink / Vibe / By owner /
Latest / Street View & 360° / Videos), so Phase 2's photo role mapping has
an evidence-based prior rather than guessing from filename. PR #80
(`brand_analyses.photo_roles`) is the recipient of that signal — the more
reliable the inputs, the lower the drift rate.

## Stack
User-level Bash + Python (matches the existing `~/.claude/scripts/apify/`
patterns). Apify MCP (already authenticated). No NERVE changes.

## Integrations
- Apify Store actor `solidcode/google-maps-photos-scraper`
  ($0.0005/photo + $0.00005 actor start, 99.4% success rate).
- Existing Apify MCP server registration (`apify` MCP).

## How to verify

1. Confirm wrapper exists and is executable:
   ```bash
   ls -la ~/.claude/scripts/apify/google-maps.sh
   ```
2. Smoke-test with the Blackbird sample input:
   ```bash
   echo '[{"photos":[{"imageUrl":"https://lh3.googleusercontent.com/p/AF1QipPyyUt2f7eyEGZp_mfQqsEyrn8RPWH4fB9mB9ba=w800-h600-k-no","photoId":"AF1QipPyyUt2f7eyEGZp_mfQqsEyrn8RPWH4fB9mB9ba","category":"by_owner"}]}]' \
   | bash ~/.claude/scripts/apify/google-maps.sh "test" /tmp/gmaps_test
   ls /tmp/gmaps_test
   ```
   Expect: `gmaps_test_yyUt2f7eyEGZ.jpg` (a real cake photo).
3. End-to-end: run `/spec-site-brief` on a fresh lead. Verify the assistant
   calls `mcp__apify__call-actor` with the solidcode actor and that the
   downloaded `gmaps_*.jpg` files appear in `photos/` before brand decode.
   Confirm Phase 2 photo_roles map references the new files.

## Known issues

- The fallback search-style URL (`https://www.google.com/maps/place/<name>+<address>`)
  occasionally resolves to the wrong place when the business name is
  generic. Prefer the canonical URL with CID when Phase 1 finds it.
  The actor returns `totalItemCount: 0` on miss — surface that as a
  silent skip, don't fabricate photos.
- Google Maps photos can overlap with IG photos when the owner cross-posts.
  The wrapper does NOT dedupe across sources — Phase 2's photo_roles map
  is where duplicate-then-`unused` decisions get recorded with drift
  tracking via PR #80.
- For non-trading businesses, the Google listing may still exist but have
  zero photos. The actor returns an empty `photos[]` and the wrapper
  exits 1; skill continues with whatever IG + Fresha photos surfaced.
