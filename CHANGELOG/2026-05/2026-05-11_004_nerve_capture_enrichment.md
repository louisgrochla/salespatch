# Capture enrichment — pitch briefs + brief reasoning + demo metadata

Pre-Phase-F2 capture hygiene. The four skills emit signal that wasn't
landing in NERVE before this PR. Without the changes here, that signal
vanishes after each chat closes and the F4 agent SDK inherits prompts
that don't durably capture their own reasoning.

## What changed

### #1 — `/lead-json` pitch playbook → NERVE

- `apps/nerve/prisma/schema.prisma` — new `PitchBrief` model
  (`pitch_briefs` table). Lead-card surface (services, pain_points,
  trust_badges, avoid_topics, opening_hours, etc.) + sales playbook
  (hook, opener, demo_moments, close_script, next_visit_reason,
  specific_objections). Indexed on (lead_id, generated_at DESC) for
  "latest playbook for X" lookups.
- `apps/nerve/prisma/migrations/17_pitch_briefs/migration.sql` — new
  table.
- `apps/nerve/src/lib/sl-mas/pitchBriefStore.ts` — store with
  `ingest`, `getById`, `latestForLead`, `listForLead`,
  `listByVertical`. Idempotent on `pitch_brief_id` =
  `<slug>-pitch-<iso_no_colons>`.
- `apps/nerve/src/app/api/ingest/pitch-brief/route.ts` — HMAC POST
  endpoint. Same `OUTCOME_INGEST_SECRET` + `verifySignature` pattern
  as the rest of `/api/ingest/*`. Auto-calls
  `businessIdentityStore.lookupOrCreate` after the upsert so the F1
  canonical row stays in sync.
- `apps/nerve/src/app/(app)/leads/[id]/page.tsx` — new "Pitch brief"
  section renders hook / opener / demo_moments / close_script /
  next_visit_reason / objections + a rep-notes line for
  pain_points / trust_badges / avoid_topics. Visible day-1 so the
  capture surfaces in the operator workflow, not just in the warehouse.
- `~/.claude/commands/lead-json.md` (user-level, not committed) —
  emits `outputs/pitch-brief.json` and POSTs it via
  `~/.claude/scripts/nerve/post-ingest.sh`. Output format adds one
  line summarising the NERVE post status.

### #5 — `/spec-site-brief` reasoning trace into metadata

- `~/.claude/skills/spec-site-brief/SKILL.md` (user-level, not
  committed) — Phase 2 (aesthetic positioning) and Phase 3
  (diagnosis) now silently capture the alternatives considered + why
  rejected. `outputs/brief.json` `metadata` field gains three required
  keys: `diagnosis_alternatives_considered`,
  `positioning_alternatives_considered`, `verdict_reasoning_trace`.
  No schema change — `site_briefs.metadata` is already JSONB.

### #3 — `/build-demo` metadata enrichment

- `~/.claude/commands/build-demo.md` (user-level, not committed) —
  `outputs/demo-artefact.json` `metadata` field gains
  `photo_classifications` (filename → role), `photos_unused_count`,
  and `nerve_consult_summary` (which strategies returned, which
  dimensions got biased by champion, which the brief overrode,
  reason for no-signal when applicable). No schema change —
  `demo_artefacts.metadata` is already JSONB.

## Why

The audit run after F1 merged surfaced three concrete capture gaps:

1. The salesperson pitch playbook (hook / opener / demo_moments /
   close_script / objections) was generated by `/lead-json`, written
   to the local submit folder, and dragged into the admin form by
   hand. None of it ever landed in NERVE — the warehouse held the
   demo HTML and the upstream research brief but not the rep-facing
   pitch script. The B1 lead_assignment_events stream captures
   outcomes (visited / pitched / sold / rejected); without the
   playbook capture, a future agent can never join "which pitch
   script wording closed vs got rejected."

2. The reasoning trace in `/spec-site-brief` (which diagnoses /
   positioning references were considered and rejected) lived only
   in the chat output. The warehouse captured the winning diagnosis
   but not the rejected alternatives. A future critic that learns
   "stop suggesting owned-audience gap for sub-2k IG bakeries"
   needs the rejected-options corpus to converge — not just the
   wins.

3. `/build-demo` photo classifications (which photo went to which
   slot) and NERVE consultation outcomes (which champion dimensions
   were applied vs overridden) lived only in the chat output. Both
   are learnable design dimensions; both vanish without persistence.

All three are pre-F4 hygiene. F4 freezes the skill prompts behind
agent SDK invocations; the longer we run skills that don't capture
their reasoning, the smaller the F4 corpus is when it lands.

## Stack
- Next.js 14 app-router route handler (HMAC, same pattern as B1-B4
  / site-brief / demo-artefact)
- Prisma 5.22 / Postgres — single new table, JSONB column for
  specific_objections + metadata
- No producer wiring changes for #3 + #5 (metadata-only additions
  to existing payloads); the migration only covers #1.

## Integrations
- New endpoint `POST /api/ingest/pitch-brief` follows the existing
  HMAC pattern. No new secrets.
- `businessIdentityStore.lookupOrCreate` integration ensures the F1
  canonical row gets updated when a pitch-brief lands first (e.g. if
  /lead-json runs without a prior /spec-site-brief). Soft FK on
  `brief_id` — null when /spec-site-brief was skipped.
- Lookups against `pitch_briefs` are now part of the /leads/[id]
  fan-out — the Promise.all there grows from 10 to 12 parallel
  queries. No latency concern at single-operator scale.

## Known issues
- The `/lead-json`, `/spec-site-brief`, and `/build-demo` skill
  markdowns live in `~/.claude/commands/` (user-level, not in this
  repo). The capture-enrichment changes in those files are listed
  above for traceability; CI cannot test them.
- PASS-verdict pitch briefs still emit a row with null playbook
  fields — intentional, so the warehouse captures the
  disqualification pattern. The /leads/[id] Pitch brief section
  hides itself when `hook` is null + `demo_moments` is empty
  (PASS case) to avoid showing an empty panel.
- #2 (lead-hunter run trace) and #4 (auto-QA pass) were
  intentionally deferred. #2 wants 10+ real runs first to design
  the schema from observed shapes; #4 is a separate skill change
  that deserves its own PR.
