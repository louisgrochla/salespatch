# /preview/[leadId] — customer-facing demo + sticky CTA

**What changed**
- `apps/sales-dashboard/src/app/preview/[leadId]/page.tsx` — new public page. Server component: looks up lead_assignment, parses `business_name` + `demo_site_domain` from notes JSON, calls `getOrCreateActiveSession` to fetch (or refresh) the active Stripe Checkout session, then renders an iframe of the demo with a sticky bottom CTA: "Go live now · £350 setup, then £25/mo →" linking to `checkout_url`. Three states: with-CTA (default), paid (assignment.status='sold' → no CTA), demo-missing (fallback).
- `apps/sales-dashboard/src/middleware.ts` — added `/preview`, `/onboarding`, `/paid`, `/api/payments/webhook`, `/api/onboarding` to PUBLIC_PATHS.
- `apps/sales-dashboard/src/components/ConditionalShell.tsx` — added the three customer-facing routes to NO_SHELL_ROUTES so they render without the salesperson AppShell.

**Why**
Step 6 of the customer payment flow handover. Customer scans the QR on their own phone; this is what they see. Sticky CTA copy is locked: `Go live now · £350 setup, then £25/mo →`. The recurring fee disclosure on the bar (not just on Checkout) is a UK consumer-law requirement.

**Stack**
Next.js 14 App Router (server component), Supabase Postgres, Stripe Checkout (hosted, full-page redirect via `<a href>` — not embedded). Inline CSS to avoid pulling in Tailwind / brand primitives client-side; preview must paint fast on a throttled connection. Mobile-first viewport (`100dvh`, `env(safe-area-inset-bottom)`).

**Integrations**
- Supabase: `lead_assignments` (notes JSON for business_name + demo_site_domain).
- Stripe: lazy session refresh via `getOrCreateActiveSession`.
- Demo HTML: served from Supabase Storage `demo-sites` bucket via the URL stored in `notes.demo_site_domain`. Iframe sandbox is `allow-scripts allow-same-origin allow-popups allow-forms`.

**How to verify**
1. Make sure migration 003 (commission_amount_pence) and a contractor + assignment + demo_site_domain are present.
2. Visit `https://salespatch.co.uk/preview/<assignment-id>` (no auth required) on a phone.
3. Demo loads inside an iframe; sticky bottom bar shows the CTA copy.
4. Tap CTA → Stripe Checkout (test card 4242…) → on success redirects to `/onboarding/<id>`.
5. After payment, revisit the same `/preview/<id>` → sticky bar should now show the "✓ Paid — your real site is being built" state instead of the CTA. (Caveat: `force-dynamic` so this updates on next request, no cache wait.)
6. Try `/preview/<bogus-id>` → 404.
7. Try a lead whose notes have no `demo_site_domain` → fallback message renders, CTA still works.

**Known issues**
- Live preview verification skipped — requires Stripe test keys, Supabase data, and a deployed `salespatch.co.uk` env. Will smoke-test in step 11 (end-to-end).
- Doc edge case: "Strip or wrap inline `<a href>` links in the demo HTML server-side." Skipped for v1. Iframe sandboxing means demo links open in the iframe, not the parent. Customer can still tap a demo's "Order now" button and get a 404 — known limitation, accepted for beta.
- Edge cache: page is `force-dynamic` for now (correctness over speed). When confident, switch to `revalidate=60` for a faster cold paint.
- Scan tracking (`POST /api/leads/:id/viewed`) not yet wired — useful signal for the contractor's iOS app ("Demo viewed 12s ago") but not on the critical path. Can land alongside step 9 if needed.
